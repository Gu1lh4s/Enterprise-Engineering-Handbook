## 1. ISO 27001 Overview

```
ISO/IEC 27001:2022 — Information Security Management Systems (ISMS)

What it certifies:
  Your organization has a systematic approach to managing information security risks.
  NOT: that you are 100% secure (impossible)
  YES: that you identify, assess, and treat risks systematically

Structure:
  Clauses 4-10 (mandatory):
    4. Context of the organization
    5. Leadership and commitment
    6. Planning (risk assessment and treatment)
    7. Support (resources, competence, awareness)
    8. Operation (implement controls, manage changes)
    9. Performance evaluation (monitoring, audit)
    10. Improvement (nonconformities, continual improvement)
  
  Annex A: 93 controls across 4 themes (select applicable controls):
    5. Organizational controls (37 controls)
    6. People controls (8 controls)
    7. Physical controls (14 controls)
    8. Technological controls (34 controls)

Why pursue ISO 27001:
  - Enterprise sales requirement (Fortune 500 vendor questionnaires)
  - GDPR accountability demonstration
  - Cyber insurance premium reduction
  - Competitive differentiation
  - Internal security maturity framework

Timeline: typically 6-18 months from start to certification
Cost: $15K-$150K depending on organization size and consultant fees
```

## 2. Information Security Management System (ISMS)

### ISMS Scope Definition

```
ISMS Scope Statement example:
  "The ISMS covers the design, development, operation, and support of [Product Name],
  including the cloud infrastructure (AWS/GCP/Azure), CI/CD pipelines, source code
  repositories, customer data processing, and the [City] office network.
  
  Excluded: [Subsidiary Name] (separate legal entity with own ISMS)"

Scope determines:
  - Which assets are in scope
  - Which risks must be assessed
  - Which Annex A controls are applicable
  - What auditors will examine

Scope boundaries to define:
  ✓ Products / services
  ✓ Locations (offices, data centers, cloud regions)
  ✓ Organizational units
  ✓ Technology platforms
  ✗ Explicitly excluded items (with justification)
```

## 3. Risk Assessment Methodology

### Risk Register

```
Risk = Threat × Vulnerability × Asset Value × Likelihood

Likelihood scale (1-5):
  1 = Rare (< once in 5 years)
  2 = Unlikely (once in 2-5 years)
  3 = Possible (once a year)
  4 = Likely (monthly)
  5 = Almost certain (weekly)

Impact scale (1-5):
  1 = Negligible (< 1K damage, no regulatory impact)
  2 = Minor (< 10K damage, minor regulatory)
  3 = Moderate (< 100K damage, reportable breach)
  4 = Major (< 1M damage, regulatory fine, reputational)
  5 = Catastrophic (> 1M damage, business threatening)

Risk score = Likelihood × Impact
  1-6:   Low → Accept
  8-12:  Medium → Monitor / mitigate
  15-25: High → Must treat (implement controls)
```

```typescript
// Risk register schema (store in Notion, Confluence, or database)
interface Risk {
  id: string                          // R-001, R-002...
  title: string                       // "Unauthorized access to customer database"
  assetId: string                     // Links to asset register
  threat: string                      // "External attacker exploits SQL injection"
  vulnerability: string               // "Web application not using parameterized queries"
  currentControls: string[]           // ["WAF", "Input validation in some endpoints"]
  likelihood: 1 | 2 | 3 | 4 | 5
  impact: 1 | 2 | 3 | 4 | 5
  inherentRisk: number                // likelihood × impact (before treatment)
  treatment: 'accept' | 'mitigate' | 'transfer' | 'avoid'
  treatmentPlan: string               // "Implement parameterized queries in all ORM calls"
  treatmentOwner: string              // "Engineering team - Sprint Q3 2025"
  residualLikelihood: 1 | 2 | 3 | 4 | 5
  residualImpact: 1 | 2 | 3 | 4 | 5
  residualRisk: number                // After controls implemented
  reviewDate: Date
}

// Risk Treatment Options:
// Mitigate: implement controls to reduce risk score
// Transfer: insurance, outsourcing (liability moves, risk stays)
// Accept:   documented decision for low risks (risk appetite)
// Avoid:    stop the activity that creates the risk
```

### Statement of Applicability (SoA)

```markdown
The Statement of Applicability is a MANDATORY document.
It lists all 93 Annex A controls and whether each is:
  - Applicable (YES/NO)
  - If NO: documented justification
  - If YES: implemented (YES/PARTIAL/NO) and evidence reference

Example SoA entry:

| Control | Title | Applicable | Implemented | Justification / Evidence |
|---------|-------|------------|-------------|--------------------------|
| 8.2 | Privileged access rights | Yes | Yes | IAM policy; quarterly access review log |
| 8.9 | Configuration management | Yes | Partial | Terraform for infra; app config management in progress |
| 7.4 | Physical security monitoring | No | N/A | All infrastructure is cloud-hosted (AWS); no physical servers |
| 5.30 | ICT readiness for business continuity | Yes | Yes | DR plan; tested quarterly; RTO/RPO in BC policy |
```

## 4. Annex A Controls Reference

### Technology Controls (most relevant for SaaS)

```
8.2  - Privileged access rights
  → Implement: PAM (CyberArk/Teleport), JIT access, quarterly reviews
  Evidence: access review log, JIT access request records

8.3  - Information access restriction
  → Implement: RBAC per role per feature, data classification
  Evidence: RBAC matrix, user access list by role

8.5  - Secure authentication
  → Implement: MFA required, strong password policy, no default creds
  Evidence: IdP config screenshots, MFA enrollment report

8.6  - Capacity management
  → Implement: HPA, monitoring, alerts on > 80% capacity
  Evidence: Grafana dashboard, capacity planning doc

8.7  - Protection against malware
  → Implement: EDR on endpoints, image scanning, runtime protection
  Evidence: Crowdstrike/Defender reports, Trivy scan results in CI

8.8  - Management of technical vulnerabilities
  → Implement: CVE scanning (Snyk/Dependabot), patch SLA, vuln register
  Evidence: Snyk reports, patch management schedule

8.15 - Logging
  → Implement: centralized logging (structured JSON, retained 90 days min)
  Evidence: Loki/CloudWatch config, retention policy

8.16 - Monitoring activities
  → Implement: SIEM rules, anomaly detection, alert runbooks
  Evidence: alert rules config, on-call runbook

8.24 - Use of cryptography
  → Implement: TLS 1.2+ only, no MD5/SHA1, AES-256 for storage
  Evidence: SSL Labs A+ report, cryptography policy doc

8.25 - Secure development lifecycle
  → Implement: SAST/DAST in CI, code review, security training
  Evidence: CI pipeline logs, Semgrep reports, security training completion

8.28 - Secure coding
  → Implement: OWASP Top 10 awareness, secure coding standards
  Evidence: code review checklist, developer security training records

8.32 - Change management
  → Implement: PR review required, staging environment, change approval
  Evidence: GitHub branch protection rules, deployment approval records
```

### Data Classification

```
Classification levels:
  PUBLIC:       Intended for public access (marketing website, open-source code)
  INTERNAL:     For employees only (internal docs, org charts, meeting notes)
  CONFIDENTIAL: Business-sensitive (source code, customer lists, financial data)
  RESTRICTED:   Highest sensitivity (credentials, PII, payment data, trade secrets)

Handling requirements per classification:
  PUBLIC:
    - May be shared externally without approval
    - No special storage requirements
  
  INTERNAL:
    - Share only with employees and approved contractors
    - Do not post on public forums/channels
  
  CONFIDENTIAL:
    - Encrypt in transit (TLS) and at rest
    - Need-to-know access controls
    - Log access to confidential systems
    - Do not store on personal devices
  
  RESTRICTED:
    - Encrypt with AES-256
    - Strict need-to-know (minimal number of people)
    - Full audit logging of all access
    - MFA required for systems holding restricted data
    - No storage on endpoints without endpoint encryption
    - Notify security team of any suspected exposure
```

## 6. Access Control

### Joiners, Movers, Leavers (JML) Process

```typescript
// Control 5.18 - Access rights (JML process)

// On new hire (JOINER):
async function onboardUser(userId: string, role: string, department: string) {
  // 1. Create identity (SSO)
  await ssoProvider.createUser(userId)
  
  // 2. Assign baseline access (role-based)
  await assignBaselineAccess(userId, role, department)
  
  // 3. Send to onboarding: require security awareness training
  await trainingSystem.enroll(userId, 'security-awareness-onboarding')
  
  // 4. Record in access registry
  await accessRegistry.create({ userId, role, startDate: new Date(), createdBy: 'HR System' })
}

// On role change (MOVER):
async function moveUser(userId: string, newRole: string, newDepartment: string) {
  const currentAccess = await accessRegistry.getByUserId(userId)
  
  // Remove old role access
  await revokeRoleAccess(userId, currentAccess.role)
  
  // Grant new role access
  await assignBaselineAccess(userId, newRole, newDepartment)
  
  await accessRegistry.update(userId, { role: newRole, department: newDepartment, changedAt: new Date() })
}

// On termination (LEAVER):
async function offboardUser(userId: string, terminationType: 'voluntary' | 'involuntary') {
  const INVOLUNTARY_IMMEDIATE = terminationType === 'involuntary'
  
  if (INVOLUNTARY_IMMEDIATE) {
    // Immediate revocation for involuntary terminations
    await ssoProvider.disableUser(userId)           // Disable SSO (kills all sessions)
    await revokeAllApiKeys(userId)
    await revokeAllSshKeys(userId)
    await removeFromAllGroups(userId)
  } else {
    // Schedule for last day
    await scheduleRevocation(userId, lastWorkingDay)
  }
  
  await accessRegistry.recordLeaver(userId, terminationType)
  await notifyIT(userId, terminationType)
}
```

### Privileged Access Review

```typescript
// Control 8.2 - Quarterly privileged access review
// Must be documented and evidenced for auditors

async function runQuarterlyAccessReview(quarter: string) {
  const privilegedAccounts = await listPrivilegedAccounts()  // admin, superuser, root
  
  const reviewReport: AccessReviewItem[] = []
  
  for (const account of privilegedAccounts) {
    const lastActivity = await getLastActivity(account.id)
    const isActive = await isStillEmployee(account.userId)
    
    reviewReport.push({
      account: account.username,
      userId: account.userId,
      role: account.role,
      lastActivity,
      stillEmployed: isActive,
      lastReviewedBy: null,  // To be filled in by reviewer
      decision: null,        // 'retain' | 'revoke' | 'modify'
      reviewDate: null,
    })
  }
  
  // Export for review → manager approves each entry
  await exportForReview(reviewReport, quarter)
  
  // After review: process decisions
  // Anyone marked 'revoke': offboard
  // Orphaned accounts (employee left): immediate revocation
}
```

## 7. Cryptography Policy

```
ISO 27001:2022 Control 8.24 - Use of cryptography

Approved algorithms:
  Symmetric encryption: AES-256-GCM (data at rest), ChaCha20-Poly1305
  Asymmetric:           RSA-4096, ECDSA P-256/P-384, Ed25519
  Hashing:              SHA-256, SHA-384, SHA-512, BLAKE2
  Password hashing:     Argon2id (primary), bcrypt cost ≥ 12 (legacy compat)
  TLS:                  1.3 (preferred), 1.2 (minimum); no TLS 1.0/1.1
  Key exchange:         ECDHE, X25519

Prohibited:
  DES, 3DES, RC4, MD5, SHA-1 for signatures, ECB mode, RSA < 2048,
  Custom/home-grown cryptographic implementations

Key Management:
  Key generation:    Use cryptographically secure RNG (crypto.getRandomValues, /dev/urandom)
  Key storage:       AWS KMS, GCP KMS, HashiCorp Vault (never in code or version control)
  Key rotation:      Annual minimum; immediate on suspected compromise
  Key escrow:        Document who can recover encrypted data if key owner unavailable
  Key destruction:   Log destruction; verify cryptographic erasure
```

## 9. Business Continuity

```
ISO 27001 Annex A 5.30 - ICT readiness for business continuity

Recovery Objectives:
  RTO (Recovery Time Objective): Maximum time to restore service
  RPO (Recovery Point Objective): Maximum data loss acceptable
  
System tiers:
  Tier 1 (Critical): RTO < 1h, RPO < 15min
    → Booking system, payment processing, authentication
  Tier 2 (Important): RTO < 4h, RPO < 1h
    → Admin portal, reporting, notifications
  Tier 3 (Deferrable): RTO < 24h, RPO < 24h
    → Batch exports, analytics, marketing integrations

Backup Strategy (3-2-1 Rule):
  3 copies of data
  2 different media types (disk + S3)
  1 offsite location (different AWS region)

DR Testing (mandatory for certification):
  - Test RTO/RPO achievement quarterly
  - Annual full DR exercise (simulate region failure)
  - Document test results and gaps
  - Update DR plan based on findings
```

```bash
# Automated backup validation
#!/bin/bash
# backup-validation.sh — run daily in CI to verify backups are restorable

set -euo pipefail

BACKUP_BUCKET="s3://backups-prod"
TEST_DB="dr-validation-$(date +%Y%m%d)"

echo "[$(date)] Starting backup validation"

# 1. Find latest backup
LATEST_BACKUP=$(aws s3 ls "${BACKUP_BUCKET}/postgres/" | sort | tail -1 | awk '{print $4}')
echo "Latest backup: $LATEST_BACKUP"

# 2. Restore to test database
aws s3 cp "${BACKUP_BUCKET}/postgres/${LATEST_BACKUP}" /tmp/backup.sql.gz
pg_restore -h localhost -U admin -d "${TEST_DB}" /tmp/backup.sql.gz

# 3. Validate row counts against expected values
BOOKING_COUNT=$(psql -h localhost -U admin -d "${TEST_DB}" -t -c "SELECT COUNT(*) FROM bookings")
if [ "$BOOKING_COUNT" -lt 1000 ]; then
  echo "ERROR: Expected at least 1000 bookings, got ${BOOKING_COUNT}"
  exit 1
fi

# 4. Cleanup test database
psql -h localhost -U admin -c "DROP DATABASE ${TEST_DB}"

echo "[$(date)] Backup validation successful ✓"
# Alert: send success metric to monitoring
curl -X POST "${MONITORING_URL}/backup-validation/success" -d "timestamp=$(date +%s)"
```

## 11. Audit and Certification Preparation

### Evidence Collection Checklist

```
Documentation (must exist and be current):
  [ ] ISMS scope statement
  [ ] Information Security Policy (signed by top management)
  [ ] Risk assessment (dated within 12 months)
  [ ] Statement of Applicability (all 93 controls addressed)
  [ ] All mandatory policies (10 minimum)
  [ ] Asset register (current)
  [ ] Supplier register with tier classification

Operational evidence (must show ongoing operation):
  [ ] Quarterly access reviews (last 2+ quarters)
  [ ] Monthly vulnerability scans (last 3+ months)
  [ ] Security awareness training completion records (last year)
  [ ] Change management records (last 3 months of PRs/approvals)
  [ ] Incident log (even if empty → shows process exists)
  [ ] Internal audit report (must complete before Stage 2)
  [ ] Management review meeting minutes

Technical evidence:
  [ ] SSL Labs A+ report
  [ ] MFA enrollment report (% of users with MFA)
  [ ] Penetration test report (< 12 months old) + remediation evidence
  [ ] SAST/DAST scan results from CI
  [ ] Backup restore test results
  [ ] DR exercise results

Common audit findings to avoid:
  ✗ Policies not signed/approved
  ✗ Risk assessment not linked to controls
  ✗ Access reviews done but not documented
  ✗ Supplier contracts missing DPA
  ✗ No evidence of security awareness training
  ✗ Internal audit not completed before Stage 2
  ✗ Management review skipped (must involve top management)
```

### Certification Timeline

```
Month 1-2: Gap Assessment
  - Current state vs. ISO 27001 requirements
  - Identify what's missing
  - Prioritize remediation

Month 3-4: Foundation
  - ISMS scope, policy suite, risk register
  - Asset inventory, Statement of Applicability
  - Start evidence collection

Month 5-8: Control Implementation
  - Technical controls (access, encryption, monitoring)
  - Operational procedures (incident response, change management)
  - Supplier reviews, DPAs

Month 9-10: Internal Audit
  - Independent review of ISMS
  - Document findings and corrective actions
  - Management review meeting

Month 11: Stage 1 Audit (Documentation Review)
  - Auditor reviews documentation
  - Issues: clarifications, minor nonconformities, major findings
  - Typical outcome: 4-8 weeks to address findings

Month 12-13: Stage 2 Audit (Implementation Audit)
  - Auditor verifies controls are operational
  - Interviews with staff (must demonstrate awareness)
  - Evidence reviews

Month 14: Certification decision
  - Certificate valid 3 years
  - Surveillance audits: years 1 and 2
  - Recertification audit: year 3
```
